# 任务分级与提示词粒度规约

本文约束 ChatGPT、Codex 任务设计者及其他 AI Agent 生成单轮开发任务提示词时的粒度选择。

核心原则：提示词长度、阅读范围、分析深度、测试范围和执行动作必须与任务实际复杂度成正比。禁止机械套用大型开发模板。

## 1. 分级前置要求

生成任务提示词前，必须先判断任务等级，再选择对应模板和上下文范围。

任务等级由实际影响范围决定，不由描述字数、用户语气或模板名称决定。无法确定时，优先按较小等级设计；只有发现明确的跨模块影响后，才允许升级等级并说明理由。

不得为了显得完整，把微小修改包装成架构任务。

## 2. S 级：微小修改

适用范围包括但不限于：

- 单个 Maven 或 npm 依赖调整。
- 单个明确 Bug 修复。
- 单个配置项修改。
- 单条 SQL 修改。
- 单个接口字段调整。
- 单个前端样式或局部交互修改。
- 单个文档错误修正。

S 级提示词通常只需要说明：

1. 修改目标。
2. 修改范围或目标文件。
3. 验证方式。
4. 提交与推送要求。

S 级任务默认禁止要求：

- 全仓扫描。
- 全架构分析。
- 阅读全部 README、专题设计和架构文档。
- 无条件读取 `PROJECT_BASELINE.md`。
- 输出架构设计方案。
- 制定全量测试矩阵。
- 分析与本次修改无关的 Docker、部署或基础设施。
- 编写与修改无关的大篇幅风险报告。

一个只修改 `pom.xml`、一个 SQL、一个配置项或一个 Vue 文件的任务，不允许生成数千字提示词，除非该文件变更确实引发跨模块、兼容性、数据或运行态风险。

## 3. M 级：中型修改

适用范围包括但不限于：

- 一个完整页面。
- 一个业务功能。
- 一个边界明确的模块内修改。
- 涉及少量前后端联动的功能。

M 级任务允许按需加入：

- 阅读 `AGENTS.md`。
- 阅读与当前阶段决策直接相关的 `PROJECT_BASELINE.md`。
- 阅读本模块相关专题文档。
- 进行模块内影响分析。
- 执行中等规模、与修改范围匹配的测试。

禁止因为任务达到 M 级，就自动升级为全仓分析或全量测试。

## 4. L 级：大型开发

适用范围包括但不限于：

- 新模块。
- 跨多个模块的业务改造。
- 数据库结构或数据迁移调整。
- 工作流、权限模型或关键集成改造。
- 影响部署方式或多个运行服务的修改。

L 级任务可以在确有必要时要求：

- 较大范围代码与依赖分析。
- 完整影响分析。
- 多层级测试矩阵。
- Docker 与部署链路分析。
- 相关文档同步。
- 风险评估和回滚设计。

即使是 L 级任务，也必须说明每项扩大阅读、分析和验证范围的具体理由，不得把“允许”理解为“全部必做”。

## 5. XL 级：架构任务

只有以下类型的任务才应使用 XL 级完整模板：

- 新系统或大型子系统设计。
- 核心架构升级。
- 跨系统数据、权限、流程或部署架构重构。
- 重大技术选型与长期基础设施调整。

XL 级才允许默认考虑全仓分析、全量依赖分析、大型测试矩阵、完整 Docker 与部署分析、全文档同步和系统级风险评估。

禁止将普通 Bug、局部页面、单模块功能或单文件修改标记为 XL 级。

## 6. 按需阅读与最小必要原则

阅读范围必须遵循最小必要原则，只读取与本次任务直接相关的内容。

- `AGENTS.md`：当任务需要了解项目结构、长期边界、禁止事项或关键入口时读取。
- `PROJECT_BASELINE.md`：当任务依赖当前阶段目标、已验收事实、最新决策、当前风险或下一轮边界时读取。
- `docs/`：只读取本次涉及模块或专题对应的文档。

禁止为了一个局部修改，要求阅读所有 README、所有专题设计、所有架构文档或扫描整个 `docs/`。

规约文件本身也应按需引用。任务提示词可以引用适用条款或要求执行者遵守仓库规约，不要求每轮复制粘贴全部规约正文。

## 7. 提示词长度参考

以下区间用于约束复杂度，不是要求凑字数：

- S：通常 50～200 字。
- M：通常 300～800 字。
- L：通常 1000～3000 字。
- XL：仅在真正的架构任务中使用完整模板，长度按必要内容确定。

超出参考区间时，任务设计者应检查是否存在重复背景、无关阅读、机械清单、过度风险展开或不必要的全量验证要求。

任务内容完整性优先于字数限制。存在关键安全、数据、兼容性或生产运行风险时，可以超出区间，但必须保留与风险直接相关的内容，不得借此恢复机械套用大型模板。

## 8. 强制禁止条款

以下行为默认禁止：

- 机械套用大型开发模板。
- 提示词长度与任务复杂度明显不成比例。
- 未说明理由就要求全仓扫描或全架构分析。
- 未说明任务依赖就强制读取 `PROJECT_BASELINE.md` 或全部项目文档。
- 对局部修改制定与影响范围无关的全量测试矩阵。
- 对不涉及运行态的任务强行展开 Docker 与部署分析。
- 用大量流程描述掩盖不清晰的修改目标、范围和验收标准。
- 为满足模板字段而编造风险、架构问题或文档更新要求。

## 9. 任务升级规则

执行过程中发现以下情况时，允许从 S 或 M 升级到更高等级：

- 实际影响超出原定文件或模块。
- 出现数据迁移、兼容性、权限、安全或生产运行风险。
- 需要修改公共基础能力或多个调用方。
- 原定最小验证无法证明修改正确。

升级前必须说明：

1. 新发现的事实。
2. 原任务等级为什么不足。
3. 需要新增的阅读、修改和验证范围。
4. 升级后的验收标准。

不得在没有新事实的情况下自行扩大任务等级和执行范围。
