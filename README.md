# AI 协作规约仓库

本仓库用于维护人与 AI Agent（主要是 ChatGPT + Codex）协作开发时的长期通用规约。

它不是具体项目代码仓库，也不是单轮任务提示词仓库。

## 仓库定位

本仓库只维护三类公开内容：

1. `constitution.md`：个人顶层开发规约，适用于所有项目。
2. `runtime.md`：AI Agent 运行规约，约束 Codex 等执行型 Agent 的完整工作方式，包括任务启动、代码修改、构建测试、Docker 运行态收尾、汇报、提交与推送。
3. `tasks/`：任务设计与模板规约。用于约束单轮任务提示词的分级、粒度、阅读范围和结构，不保存具体任务提示词。

其中 `tasks/task-sizing.md` 是生成单轮开发任务提示词前的必读规则，用于防止对微小任务机械套用大型开发模板。

## 不在本仓库维护的内容

以下内容不放在本公开仓库：

- 每个项目的具体业务代码。
- 每个项目的项目级规约。
- 每轮任务的具体 Prompt。
- 临时排查记录。
- 项目运行日志。
- 项目私有配置、密钥、证书。

具体项目内部必须自行维护项目级协作文档。本仓库只规定这些文档的职责边界、读取顺序、维护责任和验收要求，不保存各项目的具体内容。

## 项目仓库文档分工

存在远程仓库的项目，项目根目录应采用以下文档分工模式：

1. 外部规约仓库：维护通用协作规则，即本仓库的 `constitution.md`、`runtime.md` 和 `tasks/` 模板。
2. `AGENTS.md`：项目专属地图、长期边界和 Agent 入口。用于说明项目结构、技术栈、长期约束、禁止事项、关键入口和协作方式。
3. `PROJECT_BASELINE.md`：当前阶段动态基线。用于记录当前阶段目标、已验收事实、最新决策、当前风险、下一轮任务输入和验收后需要继承的状态。
4. `docs/`：已验收历史事实、架构决策、专题设计、部署说明、接口说明、运维说明和其他长期文档。

四类文档的边界不得混淆：

- 通用规则只进入外部规约仓库。
- 项目长期入口和边界进入 `AGENTS.md`。
- 当前阶段动态状态进入 `PROJECT_BASELINE.md`。
- 已验收、可长期追溯的事实和专题文档进入 `docs/`。

## 推荐使用方式

项目启动时：

1. 引用 `constitution.md`。
2. 引用 `runtime.md`。
3. 在具体项目仓库根目录建立 `AGENTS.md`。
4. 在具体项目仓库根目录建立 `PROJECT_BASELINE.md`。
5. 在具体项目仓库中建立 `docs/`，用于保存已验收历史事实、架构决策、专题设计和部署说明。

单轮任务开始时：

1. 先同步目标项目远程仓库最新代码。
2. 读取本仓库的 `constitution.md` 和 `runtime.md`。
3. 任务设计者先按 `tasks/task-sizing.md` 判断 S、M、L 或 XL 等级，并据此确定提示词粒度、阅读范围、分析深度和验证范围。
4. 根据任务等级，按最小必要原则读取具体项目仓库中的 `AGENTS.md`、`PROJECT_BASELINE.md` 和任务相关 `docs/`；不得机械全量读取。
5. 参考 `tasks/` 中与任务等级匹配的规则或模板生成本轮任务提示词。
6. 由 Codex 执行具体开发。
7. Codex 完成后按 `runtime.md` 的完整要求汇报，包括 Docker 运行态判断与处理结果。
8. ChatGPT 或人工进行验收。
9. 验收通过后，由 ChatGPT 作为上一轮任务验收者和下一轮任务设计者，明确 `PROJECT_BASELINE.md` 的更新要求；需要 Codex 落地修改时，必须纳入下一轮或本轮收尾任务并提交推送。

## 核心原则

先做对的事，再争取把事情做对。
