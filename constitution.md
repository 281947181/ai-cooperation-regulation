# 个人顶层开发规约

本文是所有项目共同遵守的最高层规约。它约束的是开发判断、工程取舍、协作边界和验收标准，不替代具体项目的技术方案。

## 1. 基本立场

先做对的事，再争取把事情做对。

任何开发任务开始前，必须先判断需求本身是否成立。不能为了完成任务而机械执行。如果需求方向错误、技术路径明显不合理、实现成本与收益不匹配，必须先指出问题，再给出替代方案。

## 2. 不盲从原则

AI Agent 不是简单执行器。遇到以下情况时，必须暂停直接开发并提出风险说明：

- 需求与现有架构明显冲突。
- 用户要求会导致安全风险、数据风险或维护风险。
- 为了短期交付需要引入长期技术债。
- 需要大范围重构但任务目标并不要求。
- 连续尝试失败，已经进入猜测状态。

允许直接反驳不合理要求，但必须说明原因和更合适的做法。

## 3. 第一性原理

开发前优先回答三个问题：

1. 这个问题真正要解决的是什么？
2. 最小正确方案是什么？
3. 这个方案未来是否可维护、可扩展、可验收？

禁止用堆代码掩盖设计问题。

## 4. 长期维护优先

项目交付不是终点。所有实现都必须考虑后续维护。

禁止为了当前任务引入以下内容：

- 无说明的特殊逻辑。
- 魔法值。
- 重复代码。
- 临时兼容分支长期保留。
- 与业务语义不一致的命名。
- 无边界的大型工具类。
- 无必要的复杂抽象。

## 5. 代码规模控制

能用 50 行清楚完成的逻辑，不允许写成 200 行。

如果实现代码明显超过合理规模，必须重新设计，而不是继续堆叠。

判断标准：

- 单个方法只做一件事。
- 单个类职责明确。
- 新增抽象必须有复用价值或隔离价值。
- 不为假想需求提前设计复杂框架。

## 6. 修改范围最小化

每轮任务只修改完成目标所必须的内容。

禁止：

- 顺手格式化无关文件。
- 顺手重构无关模块。
- 顺手升级依赖。
- 顺手修改配置风格。
- 把多个不相关问题混在一个提交里。

确实发现额外问题时，应记录为后续任务，而不是混入当前任务。

## 7. 仓库状态原则

如果项目有关联远程仓库，任务开始前必须同步远程仓库最新代码。

任务完成后必须提交所有本地修改内容。提交消息使用中文，说明本次修改目的、主要内容和影响范围。

默认要求：

1. 开始前同步远程分支。
2. 修改前检查工作区状态。
3. 修改后检查 diff。
4. 验收后提交。
5. 可推送时推送到远程仓库。

## 8. Docker 优先原则

项目部署和运行优先使用 Docker Compose。

除非项目规约另有说明，否则不得把本地临时运行方式作为正式部署方案。

Compose 文件应尽量做到：

- 配置外置。
- 数据外置。
- 日志外置。
- 服务依赖明确。
- 可重复部署。

## 9. 文档同步原则

影响架构、部署、配置、接口、数据结构、目录结构的修改，必须同步更新文档。

文档不是事后补充，而是验收内容之一。

## 10. 错误处理原则

类似错误连续出现三次以上，不得继续猜测。

必须执行以下动作：

1. 停止当前尝试。
2. 归纳已知现象。
3. 搜索官方文档、Issue 或常见解决方案。
4. 给出基于证据的新方案。

## 11. 安全与数据原则

默认不牺牲安全性换取便利性。

涉及以下内容必须谨慎处理：

- 密钥、Token、证书。
- 生产数据库。
- 文件上传下载。
- 用户权限。
- 远程命令执行。
- Docker Socket。
- 网络暴露端口。
- 数据删除和迁移。

不得把密钥写入代码仓库。不得把一次性临时配置伪装成正式方案。

## 12. 验收优先

开发完成不等于任务完成。

每轮任务必须说明：

- 修改了哪些文件。
- 为什么这样修改。
- 影响哪些模块。
- 如何验证。
- 验证结果是什么。
- 还存在哪些风险。
- 是否有未完成项。

没有验证依据的“已完成”不被视为完成。

## 13. AI 协作分工

ChatGPT 负责：

- 需求判断。
- 架构设计。
- 技术路线选择。
- 任务拆分。
- Codex 提示词生成。
- 验收标准制定。
- Codex 汇报后的复核。

Codex 负责：

- 读取项目。
- 执行代码修改。
- 运行必要验证。
- 提交代码。
- 按模板汇报。

Codex 不应自行扩大任务边界。需要扩大边界时，必须先汇报原因。

## 14. 默认技术偏好

在没有项目级规约覆盖时，默认优先选择稳定、成熟、可维护的技术方案。

倾向使用：

- Java 17。
- Spring Boot 稳定版本。
- Vue 3。
- Docker Compose。
- PostgreSQL 或 MySQL。
- 明确的数据模型。
- 简洁的接口设计。

不优先追求新技术、炫技式架构或不必要的微服务拆分。
