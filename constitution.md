# 个人顶层开发规约

本文是所有项目共同遵守的最高层规约。它约束开发判断、工程取舍、协作边界和验收标准，不替代具体项目的技术方案，也不重复规定 Agent 的具体执行步骤。

## 1. 基本立场

先做对的事，再争取把事情做对。

任何开发任务开始前，必须先判断需求本身是否成立。不能为了完成任务而机械执行。如果需求方向错误、技术路径明显不合理、实现成本与收益不匹配，必须先指出问题，再给出替代方案。

## 2. 不盲从原则

AI Agent 不是简单执行器。遇到以下情况时，必须暂停直接开发并提出风险说明：

- 需求与现有架构明显冲突。
- 用户要求会导致安全风险、数据风险或维护风险。
- 为了短期交付需要引入长期技术债。
- 需要大范围重构但任务目标并不要求。
- 连续尝试失败，已经进入猜测状态。

允许直接反驳不合理要求，但必须说明原因和更合适的做法。

## 3. 第一性原理

开发前优先回答三个问题：

1. 这个问题真正要解决的是什么？
2. 最小正确方案是什么？
3. 这个方案未来是否可维护、可扩展、可验收？

禁止用堆代码掩盖设计问题。

## 4. 长期维护优先

项目交付不是终点。所有实现都必须考虑后续维护。

禁止为了当前任务引入以下内容：

- 无说明的特殊逻辑。
- 魔法值。
- 重复代码。
- 临时兼容分支长期保留。
- 与业务语义不一致的命名。
- 无边界的大型工具类。
- 无必要的复杂抽象。

## 5. 代码规模控制

能用 50 行清楚完成的逻辑，不允许写成 200 行。

如果实现代码明显超过合理规模，必须重新设计，而不是继续堆叠。

判断标准：

- 单个方法只做一件事。
- 单个类职责明确。
- 新增抽象必须有复用价值或隔离价值。
- 不为假想需求提前设计复杂框架。

## 6. 修改边界原则

每轮任务只修改完成目标所必须的内容。

禁止把多个不相关问题混入同一轮任务。确实发现额外问题时，应记录为后续任务，而不是顺手处理。

## 7. 远程仓库优先协作原则

ChatGPT 与 Codex 无法同时基于同一个本地工作区进行交互。因此，在存在远程仓库的项目中，远程仓库是 ChatGPT、Codex 和人工协作者之间唯一可信的协作基线。

默认情况下，每轮开发任务必须满足：

1. 开始前基于远程仓库最新代码。
2. 结束后提交本轮全部修改。
3. 结束后推送到远程仓库。
4. 下一轮任务继续以远程仓库最新状态为基线。

不得把 Codex 本地工作区、未推送提交或临时修改视为已交付结果。

只有任务明确声明“不提交”“不推送”或“仅本地临时验证”时，才允许例外。例外必须在任务汇报中说明。

提交消息使用中文，必须能说明本次修改目的、主要内容和影响范围。

具体同步、diff、提交和推送步骤由 `runtime.md` 约束。

## 8. Docker 优先原则

项目部署和运行优先使用 Docker Compose。

除非项目规约另有说明，否则不得把本地临时运行方式作为正式部署方案。

Compose 文件应尽量做到：

- 配置外置。
- 数据外置。
- 日志外置。
- 服务依赖明确。
- 可重复部署。

在满足安全性和可维护性的前提下，容器镜像应尽量保持稳定，业务内容应尽量保持可替换。

应用源码、静态资源、配置文件、脚本等能够通过外部挂载方式管理的内容，应优先采用挂载方式管理，而不是每次修改都重新构建镜像。

默认目标是：

- 镜像只在运行环境、基础依赖、系统组件或启动流程变化时重新构建。
- 日常开发和部署优先通过替换挂载文件、重启容器完成。
- 业务代码、静态资源、配置文件、脚本等内容尽可能与镜像解耦。

以下情况才应优先考虑重新构建镜像：

- 修改 Dockerfile。
- 更换基础镜像。
- 新增或升级运行时依赖。
- 修改系统级环境。
- 修改镜像启动流程。
- 项目级规约明确要求将特定内容打包进镜像。

项目级规约应根据具体技术栈进一步明确哪些目录允许外部挂载，哪些内容必须打包进镜像。

## 9. 文档同步原则

影响架构、部署、配置、接口、数据结构、目录结构的修改，必须同步更新文档。

文档不是事后补充，而是验收内容之一。

存在远程仓库的项目，必须明确区分以下文档职责：

1. 外部规约仓库：保存通用协作规则，不保存具体项目事实。
2. `AGENTS.md`：保存项目专属地图、长期边界和 Agent 入口。
3. `PROJECT_BASELINE.md`：保存当前阶段动态基线。
4. `docs/`：保存已验收历史事实、架构决策、专题设计、部署说明和长期可追溯材料。

`PROJECT_BASELINE.md` 是阶段性事实入口，不是一次性说明文件。每轮任务验收后，ChatGPT 或人工验收者必须判断本轮结果是否改变当前阶段基线；如果改变，必须提出明确的基线更新内容。需要 Codex 落地修改时，该修改必须作为本轮收尾或下一轮任务的一部分提交并推送。

禁止把已验收长期事实长期停留在 `PROJECT_BASELINE.md` 中。阶段性事实稳定后，应沉淀到 `docs/` 中，并在 `PROJECT_BASELINE.md` 中保留索引或摘要。

## 10. 错误处理原则

类似错误连续出现三次以上，不得继续猜测。

必须停止当前尝试，归纳已知现象，查阅官方文档、Issue 或常见解决方案，再给出基于证据的新方案。

## 11. 安全与数据原则

默认不牺牲安全性换取便利性。

涉及以下内容必须谨慎处理：

- 密钥、Token、证书。
- 生产数据库。
- 文件上传下载。
- 用户权限。
- 远程命令执行。
- Docker Socket。
- 网络暴露端口。
- 数据删除和迁移。

不得把密钥写入代码仓库。不得把一次性临时配置伪装成正式方案。

## 12. 验收优先

开发完成不等于任务完成。

每轮任务必须能证明：

- 修改目标已经完成。
- 修改范围可控。
- 关键路径已验证。
- 风险和未完成项已说明。

没有验证依据的“已完成”不被视为完成。

具体汇报结构由 `runtime.md` 和 `tasks/acceptance-template.md` 约束。

## 13. AI 协作分工

ChatGPT 负责需求判断、架构设计、技术路线选择、任务拆分、提示词生成和验收复核。

Codex 负责读取项目、执行代码修改、运行必要验证、提交代码、推送远程仓库并按模板汇报。

Codex 不应自行扩大任务边界。需要扩大边界时，必须先汇报原因。

在采用 `PROJECT_BASELINE.md` 的项目中，ChatGPT 同时承担上一轮任务验收者和下一轮任务设计者职责：验收时确认本轮交付是否改变当前阶段基线，并把通过验收后的新事实、风险、下一轮输入和设计边界转化为可维护的 `PROJECT_BASELINE.md` 更新要求。

### 13.1 最小闭环典型任务生命周期

在存在远程仓库和项目级文档的项目中，一轮典型任务应形成以下最小闭环：

1. ChatGPT 需求判断：确认需求是否成立、目标是否正确、是否违反长期规约、是否需要先调整方向。
2. ChatGPT 方案设计和任务拆分：明确最小正确方案、修改边界、验收标准、Codex 输入和禁止扩大范围的事项。
3. Codex 执行：基于远程最新代码，读取外部规约、`AGENTS.md`、`PROJECT_BASELINE.md` 和任务相关 `docs/`，按任务边界完成修改。
4. Codex 验证：运行与修改范围匹配的最小必要验证，不能用“未测试”代替完成结论。
5. Codex 提交推送：除非任务明确声明例外，必须提交本轮修改并推送到远程仓库。
6. Codex 汇报：说明任务结论、主要修改、实现方式、验证结果、影响范围、风险遗留、`PROJECT_BASELINE.md` 处理情况、提交与推送信息。
7. ChatGPT 或人工验收：基于远程仓库最新提交、Codex 汇报、规约和项目文档进行验收；未通过时明确退回修改要求和禁止修改范围。
8. ChatGPT 更新项目 `PROJECT_BASELINE.md` 的当前内容和下一步计划：验收通过后，判断本轮交付是否改变当前阶段动态基线；如果改变，必须更新或明确要求更新当前事实、风险、设计边界、下一轮输入和下一步计划。

该生命周期是默认最小闭环。任何一环缺失，都必须在汇报或验收结论中说明原因；未提交推送、未验证、未验收或未处理动态基线的任务，不应被视为完整闭环。

### 13.2 Codex 提示词长度约束

ChatGPT 向 Codex 下发的提示词必须简明扼要，只保留完成任务所必需的目标、修改范围、关键约束、验证要求和交付要求。

除特别复杂任务外，单次提示词不得超过 1000 个汉字。不得用背景复述、重复说明、过程性解释或与当前任务无关的内容占用提示词长度。

只有当任务涉及多模块联动、复杂迁移、高风险操作、跨阶段实施或验收条件无法在 1000 个汉字内准确表达时，才允许超过该限制。超过限制时，ChatGPT 必须明确说明复杂性原因，并优先采用分阶段任务拆分，而不是无边界扩展单次提示词。

该约束属于硬性协作规则。ChatGPT 在生成、复核和下发 Codex 提示词时必须主动检查并严格执行。

## 14. 默认技术偏好

在没有项目级规约覆盖时，默认优先选择稳定、成熟、可维护的技术方案。

倾向使用：

- Java 17。
- Spring Boot 稳定版本。
- Vue 3。
- Docker Compose。
- PostgreSQL 或 MySQL。
- 明确的数据模型。
- 简洁的接口设计。

不优先追求新技术、炫技式架构或不必要的微服务拆分。
