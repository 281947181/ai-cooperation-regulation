# Docker 运行态收尾规约

本文是 `runtime.md` 的强制补充规约，适用于 Codex 等执行型 Agent 完成代码、配置、脚本或部署相关任务后的运行态收尾。

## 1. 适用原则

每轮任务结束前，Codex 必须判断本轮修改是否影响 Docker 容器中的实际运行内容。

该判断是每轮任务的必备动作，不得省略。即使最终结论为“不涉及 Docker 运行态”，也必须在任务汇报中明确说明判断结果和依据。

以下任一情况视为影响 Docker 运行态：

1. 修改了 Docker Compose、Dockerfile、镜像构建参数或容器启动参数。
2. 修改了宿主机绑定挂载到容器中的代码、静态资源、配置、脚本、证书或其他文件。
3. 修改了构建产物，并且该产物通过挂载、复制或镜像构建进入容器。
4. 修改了需要容器内进程重新加载、重新启动或重新创建后才能生效的内容。
5. 修改了数据库迁移、初始化脚本、中间件配置或其他由容器承载的运行依赖。

## 2. 强制收尾动作

确认影响 Docker 运行态后，Codex 必须完成以下动作，除非任务明确声明仅修改代码、不允许操作运行环境：

1. 识别受影响的 Compose 项目、服务、容器、挂载路径和生效方式。
2. 将本轮修改产生的代码、配置或构建产物更新到实际运行环境所使用的位置。
3. 根据文件进入容器的方式执行正确的生效动作：
   - 绑定挂载文件：确认宿主机实际挂载源已更新；如果应用不会自动重新加载，重启对应服务。
   - 镜像内文件或 `COPY` 进入镜像的文件：重新构建镜像，并重新创建对应服务；不得只执行普通重启。
   - 命名卷或数据卷中的运行文件：先确认数据归属和覆盖风险，再按项目部署说明更新；禁止未经确认覆盖持久化数据。
   - Compose、环境变量、端口、网络、健康检查或启动命令变化：先执行 `docker compose config`，再重新创建对应服务。
4. 只处理本轮受影响的服务。禁止为了省事执行无差别的全量容器重启、全量删除或全量重建。
5. 执行运行态验证，至少包括：
   - `docker compose ps` 或等价命令确认服务状态。
   - 检查受影响服务最近日志，默认使用 `docker compose logs 服务名 --tail=100`。
   - 项目存在健康检查时，确认健康状态通过。
   - 项目存在接口、页面、端口或业务级验收入口时，执行与本轮修改直接相关的最小冒烟验证。
6. 确认运行容器使用的是本轮最新代码、配置或构建产物，禁止仅凭“容器已重启”判断变更已经生效。

## 3. 推荐命令顺序

根据项目实际情况选择最小必要命令，典型顺序如下：

```bash
docker compose config
docker compose build 受影响服务
docker compose up -d --no-deps 受影响服务
docker compose ps
docker compose logs 受影响服务 --tail=100
```

对于仅绑定挂载且应用支持热加载的场景，可以不重启，但必须验证容器读取到最新文件且业务行为已经生效。

对于仅需重启进程即可生效的场景，可以执行：

```bash
docker compose restart 受影响服务
docker compose ps
docker compose logs 受影响服务 --tail=100
```

不得机械套用命令。必须先判断是“更新挂载文件”“重启服务”还是“重建镜像并重新创建服务”。

## 4. 无法执行时的处理

出现以下情况时，可以不直接操作容器，但不得声称运行态验收完成：

1. 当前环境没有 Docker 或无容器操作权限。
2. 当前会话无法访问部署主机。
3. 任务明确禁止修改运行环境。
4. 重启或重建可能影响生产业务，且未获得操作授权。
5. 项目部署信息不足，无法可靠识别受影响服务或挂载路径。

此时必须在汇报中说明：

1. 未执行的具体动作。
2. 未执行原因。
3. 受影响的服务和文件。
4. 建议由人工执行的准确命令或操作步骤。
5. 当前结果只能视为代码或构建层完成，不能视为部署与运行态完成。

## 5. 汇报要求

Codex 每轮代码任务汇报必须增加“Docker 运行态处理”部分，至少说明：

1. 本轮是否涉及 Docker 运行态。
2. 受影响的服务、容器和挂载或镜像路径。
3. 更新文件、构建镜像、重启或重新创建服务的实际动作。
4. 执行过的验证命令和结果。
5. 未执行项、风险和人工后续动作。

如果本轮不涉及 Docker，也必须明确写明“不涉及 Docker 运行态，无需更新挂载文件或重启容器”，并说明判断依据。

## 6. 完成状态判定

对于影响 Docker 运行态的任务，只有同时满足以下条件，才可以汇报“任务完成”：

1. 代码或配置修改完成。
2. 构建和测试通过，或已明确说明无法执行的原因。
3. 实际运行环境所使用的文件或镜像已经更新。
4. 受影响服务已按正确方式重新加载、重启或重新创建。
5. 容器状态、日志、健康检查和最小业务冒烟验证通过。
6. 修改已按规约提交并推送远程仓库。

未完成运行态更新和验证时，只能汇报“代码修改完成，Docker 运行态未完成”，不得把两者混为同一完成状态。
