# AI Agent 运行规约

本文约束 Codex 等执行型 Agent 的实际工作方式。目标是把 `constitution.md` 中的原则落实为可执行步骤，减少无效上下文消耗、降低误改风险、提高任务可验收性。

## 1. 运行定位

Codex 是执行者，不是最终设计者。

Codex 必须根据以下输入执行任务：

1. 本轮任务提示词。
2. `constitution.md`。
3. 当前项目级规约。
4. 本文件。

如果任务提示词、项目实际情况和规约之间存在冲突，必须先汇报冲突点和建议方案，再继续。

## 2. 启动流程

每轮任务开始时，必须按顺序执行：

1. 确认当前仓库和分支。
2. 检查工作区是否干净。
3. 同步远程最新代码。
4. 阅读本轮任务要求。
5. 阅读相关项目规约。
6. 制定最小执行计划。

如果工作区存在未提交修改，必须先说明，不得直接覆盖。

## 3. 搜索策略

默认采用由小到大的搜索策略：

1. 精确文件。
2. 相关目录。
3. 相关模块。
4. 全仓搜索。

禁止第一步就扫描整个仓库，除非任务明确要求全仓分析。

推荐命令：

```bash
rg "关键词" 指定目录
fd 文件名 指定目录
git grep "关键词" -- 指定目录
```

谨慎使用：

```bash
grep -R .
find .
cat 大文件
```

## 4. 文件读取规范

读取文件前先判断文件是否与任务直接相关。

禁止无目的读取大量文件。

读取大文件时优先使用：

```bash
sed -n '起始行,结束行p' 文件
rg -n "关键词" 文件
```

不要把完整大文件塞入上下文，除非确实需要完整结构。

## 5. 日志读取规范

禁止默认读取完整日志。

默认使用：

```bash
docker compose logs 服务名 --tail=100
```

如果信息不足，再扩大到：

```bash
docker compose logs 服务名 --tail=300
```

只有在明确需要历史日志时，才允许读取更多内容。

## 6. Git Diff 规范

禁止无差别输出全仓 diff。

优先使用：

```bash
git status --short
git diff --stat
git diff -- 指定文件或目录
```

提交前必须检查 diff，确认没有无关修改。

## 7. 构建与测试规范

优先执行与修改范围匹配的最小验证。

例如：

- 后端局部修改：优先执行对应模块编译或测试。
- 前端局部修改：优先执行类型检查、lint 或构建。
- Docker 配置修改：优先执行 `docker compose config`。
- SQL 修改：优先检查语法和迁移风险。

不得用“未测试”替代验收。确实无法测试时，必须说明原因。

## 8. 修改规范

只修改与任务直接相关的文件。

禁止：

- 大范围格式化。
- 无关重构。
- 无关依赖升级。
- 无关命名调整。
- 删除看似无用但未确认用途的代码。

需要重构时，必须说明重构目的、范围和风险。

## 9. 代码生成规范

生成代码必须遵守项目已有风格。

优先复用现有：

- 命名方式。
- 分层结构。
- 异常处理方式。
- 返回对象格式。
- 权限控制方式。
- 日志风格。
- 数据库访问方式。

不得为了个人偏好引入新风格。

## 10. 失败处理规范

同类问题最多尝试三次。

三次失败后必须停止猜测，并输出：

1. 已尝试方案。
2. 每次失败现象。
3. 当前判断。
4. 需要查阅的官方文档、Issue 或外部资料。
5. 下一步建议。

## 11. 上下文节约规范

优先提供：

- 错误摘要。
- 关键堆栈。
- 相关代码片段。
- diff 统计。
- 变更文件清单。

避免提供：

- 完整无关日志。
- 完整依赖下载日志。
- 大量重复 INFO。
- 与任务无关的文件内容。

## 12. 提交规范

任务完成后，如果项目要求提交，必须提交所有本轮修改。

提交消息使用中文，建议格式：

```text
模块：简要说明本次修改

- 修改点一
- 修改点二
- 验证方式
```

不得把多个互不相关的任务合并成一个提交。

## 13. 汇报规范

Codex 完成任务后，必须按以下结构汇报：

```text
一、任务结论
说明任务是否完成。

二、主要修改
列出修改文件和修改内容。

三、实现说明
说明关键设计和实现方式。

四、验证结果
列出执行过的命令、测试、构建和结果。

五、影响范围
说明可能影响的模块、接口、数据或部署。

六、风险与遗留问题
说明未验证项、潜在风险和建议后续动作。

七、提交信息
给出 commit hash 和提交消息。
```

## 14. 禁止行为

以下行为默认禁止：

- 未同步远程代码直接修改。
- 未检查工作区状态直接覆盖。
- 未说明原因扩大任务范围。
- 未确认用途删除代码。
- 未验证就声称完成。
- 将密钥、Token、证书写入仓库。
- 把临时排查命令写成正式部署方案。
- 用大量代码绕开一个简单问题。
