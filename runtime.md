# AI Agent 运行规约

本文约束 Codex 等执行型 Agent 的实际工作方式。目标是把 `constitution.md` 中的原则落实为可执行步骤，减少无效上下文消耗、降低误改风险、提高任务可验收性。

## 1. 运行定位

Codex 是执行者，不是最终设计者。

Codex 必须根据以下输入执行任务：

1. 本轮任务提示词。
2. `constitution.md`。
3. `runtime.md`。
4. 当前项目仓库根目录的 `AGENTS.md`。
5. 当前项目仓库根目录的 `PROJECT_BASELINE.md`。
6. 当前项目仓库 `docs/` 中与任务直接相关的已验收文档。

如果任务提示词、项目实际情况和规约之间存在冲突，必须先汇报冲突点和建议方案，再继续。

## 2. 模型资源调度规范

AI Agent 在项目设计、规划、开发和排障过程中，必须动态判断当前任务应使用的模型规模、推理强度和响应速度。

默认配置如下：

1. 模型：当前可用的最新模型。
2. 推理强度：Medium。
3. 响应速度：Standard。

除非用户明确要求快速得到粗略结果，否则禁止默认使用 Fast / 快速响应模式。

### 2.1 决策优先级

模型资源调度必须遵循以下优先级：

1. 正确性。
2. 可维护性。
3. 可验收性。
4. Token 成本。
5. 响应速度。

不得为了节省 Token 或追求速度，牺牲关键设计、架构判断、复杂排障和验收质量。

### 2.2 升级到 High 推理的条件

出现以下任一情况时，AI Agent 必须主动建议切换到 High 推理，不得继续在 Medium 推理下勉强输出完整方案：

1. 系统架构设计，例如微服务拆分、模块边界、数据模型、权限模型、流程模型、通信协议、部署架构、AI Agent 工作流设计。
2. 重大技术选型，例如数据库、中间件、三维引擎、工业通信协议、国产化迁移方案、长期基础框架选型。
3. 首次制定大型项目方案，例如 PLM、SCADA、MES、CI/CD 平台、私有化部署平台、工业数据平台。
4. 复杂 Bug 排障，例如涉及并发、事务、锁、JVM、网络协议、容器网络、多系统联调、数据库迁移、流程引擎状态异常。
5. 大规模代码审查或重构，例如跨模块重构、1000 行以上 diff、安全审计、性能分析、接口兼容性审查。
6. 同类问题已经连续尝试三次仍未解决，或连续三轮讨论仍无法形成稳定结论。

建议升级时，AI Agent 应先停止深入展开，并明确输出：

```text
当前任务已触发 High 推理建议条件。
建议配置：最新模型 + High Reasoning + Standard Speed。
原因：说明触发条件和风险。
当前配置下不继续输出完整设计，避免形成低置信度方案。
```

### 2.3 保持 Medium 推理的条件

以下任务默认保持 Medium 推理：

1. 常规 CRUD 开发。
2. 接口开发与字段调整。
3. SQL 编写与普通查询优化。
4. Vue 页面、表单、列表、弹窗等常规前端开发。
5. Spring Boot 常规 Controller、Service、Mapper、DTO 修改。
6. Docker Compose、Nginx、应用配置的常规调整。
7. 单元测试、接口联调、局部脚本编写。
8. 文档整理、用户手册、部署说明的常规补充。

Medium 是默认工作档位，不需要每次额外说明。

### 2.4 可以降到 Low 推理的条件

只有在任务明显属于机械重复劳动时，才允许建议使用 Low 推理，例如：

1. 格式转换。
2. 变量批量改名。
3. Markdown 整理。
4. 简单翻译。
5. 日志摘录和排版。
6. 简单配置文件生成。
7. Excel、CSV 等结构化文本的低风险整理。

不得在架构设计、复杂排障、关键代码审查、数据库迁移、权限设计、流程设计中建议 Low 推理。

### 2.5 响应速度策略

默认使用 Standard。

只有用户明确表达以下意图时，才允许使用 Fast：

1. “先给个大概思路”。
2. “不用太准确”。
3. “快速判断一下”。
4. “赶紧给我一个初步结论”。

如果用户没有明确要求快速响应，则不得默认选择 Fast。

### 2.6 Codex 自动切换边界

Codex 不能仅凭 Prompt 自动切换本次请求的模型、推理强度或响应速度。

原因是模型、推理强度和响应速度属于请求参数，必须在模型开始执行前确定；Prompt 只能影响当前模型在既定参数下的行为，不能修改本次请求参数。

因此，Codex 在判断当前配置不匹配任务复杂度时，应执行以下动作：

1. 停止继续深入设计或大范围修改。
2. 明确说明当前任务建议使用的模型、推理强度和响应速度。
3. 说明继续使用当前配置的风险。
4. 等待用户手动切换，或由外层 Router / 自动化调度器重新发起任务。

如果未来运行环境支持可编程 Router、Agent Workflow、MCP Hook 或 API 参数调度，可以由外层调度器根据本规约自动选择模型和推理强度；但 Codex 本身不得假装已经完成自动切换。

## 3. 启动流程

每轮任务开始时，必须按顺序执行：

1. 确认当前仓库和分支。
2. 检查工作区是否干净。
3. 执行 `git fetch` 获取远程最新状态。
4. 执行 `git pull --ff-only` 同步当前分支。
5. 阅读本轮任务要求。
6. 阅读外部规约仓库中的 `constitution.md` 和 `runtime.md`。
7. 阅读当前项目根目录的 `AGENTS.md`。
8. 阅读当前项目根目录的 `PROJECT_BASELINE.md`。
9. 阅读当前项目 `docs/` 中与本轮任务直接相关的文档。
10. 制定最小执行计划。
11. 按本文件“模型资源调度规范”判断当前配置是否匹配任务复杂度。

如果工作区存在未提交修改，必须先说明，不得直接覆盖。

如果 `git pull --ff-only` 失败，必须停止开发并汇报原因，不得基于分叉或未知状态继续修改。

如果项目缺少 `AGENTS.md` 或 `PROJECT_BASELINE.md`，且本轮任务依赖项目级协作边界或当前阶段状态，必须先汇报缺口。除非任务明确要求补建这些文件，否则不得自行编造项目基线。

## 4. 项目文档读取与维护规范

项目仓库采用以下文档职责分工：

1. `AGENTS.md`：项目专属地图、长期边界和 Agent 入口。
2. `PROJECT_BASELINE.md`：当前阶段动态基线。
3. `docs/`：已验收历史事实、架构决策、专题设计、部署说明和长期可追溯材料。

读取顺序必须遵循：先读 `AGENTS.md`，再读 `PROJECT_BASELINE.md`，最后按任务需要读取 `docs/` 中的相关文档。不得用全量扫描 `docs/` 替代精准读取。

`PROJECT_BASELINE.md` 的维护规则：

1. 每轮任务完成后，必须判断本轮结果是否改变当前阶段基线。
2. 如果改变，必须更新或提出更新 `PROJECT_BASELINE.md`。
3. 更新内容应只记录当前阶段需要继承的事实、边界、风险、下一轮输入和待验收事项。
4. 已经稳定并通过验收的长期事实，应沉淀到 `docs/`，`PROJECT_BASELINE.md` 中只保留摘要或索引。
5. 不得把临时排查过程、未验证猜测、一次性日志和未验收结论写入 `PROJECT_BASELINE.md`。

当 ChatGPT 已在验收结论中给出 `PROJECT_BASELINE.md` 更新要求时，Codex 必须按要求落地修改；如果认为更新内容与代码实际不一致，必须先汇报冲突，不得擅自改写基线事实。

## 5. 搜索策略

默认采用由小到大的搜索策略：

1. 精确文件。
2. 相关目录。
3. 相关模块。
4. 全仓搜索。

禁止第一步就扫描整个仓库，除非任务明确要求全仓分析。

推荐命令：

```bash
rg "关键词" 指定目录
fd 文件名 指定目录
git grep "关键词" -- 指定目录
```

谨慎使用：

```bash
grep -R .
find .
cat 大文件
```

## 6. 文件读取规范

读取文件前先判断文件是否与任务直接相关。

禁止无目的读取大量文件。

读取大文件时优先使用：

```bash
sed -n '起始行,结束行p' 文件
rg -n "关键词" 文件
```

不要把完整大文件塞入上下文，除非确实需要完整结构。

## 7. 日志读取规范

禁止默认读取完整日志。

默认使用：

```bash
docker compose logs 服务名 --tail=100
```

如果信息不足，再扩大到：

```bash
docker compose logs 服务名 --tail=300
```

只有在明确需要历史日志时，才允许读取更多内容。

## 8. Git Diff 规范

禁止无差别输出全仓 diff。

优先使用：

```bash
git status --short
git diff --stat
git diff -- 指定文件或目录
```

提交前必须检查 diff，确认没有无关修改。

## 9. 构建与测试规范

优先执行与修改范围匹配的最小验证。

例如：

- 后端局部修改：优先执行对应模块编译或测试。
- 前端局部修改：优先执行类型检查、lint 或构建。
- Docker 配置修改：优先执行 `docker compose config`。
- SQL 修改：优先检查语法和迁移风险。

不得用“未测试”替代验收。确实无法测试时，必须说明原因。

## 10. 修改规范

只修改与任务直接相关的文件。

禁止：

- 大范围格式化。
- 无关重构。
- 无关依赖升级。
- 无关命名调整。
- 删除看似无用但未确认用途的代码。

需要重构时，必须说明重构目的、范围和风险。

## 11. 代码生成规范

生成代码必须遵守项目已有风格。

优先复用现有：

- 命名方式。
- 分层结构。
- 异常处理方式。
- 返回对象格式。
- 权限控制方式。
- 日志风格。
- 数据库访问方式。

不得为了个人偏好引入新风格。

## 12. 失败处理规范

同类问题最多尝试三次。

三次失败后必须停止猜测，并输出：

1. 已尝试方案。
2. 每次失败现象。
3. 当前判断。
4. 需要查阅的官方文档、Issue 或外部资料。
5. 下一步建议。

如果三次失败的问题属于复杂 Bug、架构判断或重大技术选型，必须同时触发 High 推理建议，不得继续在当前配置下反复试错。

## 13. 上下文节约规范

优先提供：

- 错误摘要。
- 关键堆栈。
- 相关代码片段。
- diff 统计。
- 变更文件清单。
- `PROJECT_BASELINE.md` 是否需要更新的结论。

避免提供：

- 完整无关日志。
- 完整依赖下载日志。
- 大量重复 INFO。
- 与任务无关的文件内容。

## 14. 提交与推送规范

除非任务明确声明“不提交”“不推送”或“仅本地临时验证”，否则任务完成后必须提交并推送远程仓库。

默认执行顺序：

```bash
git status --short
git diff --stat
git add 本轮修改文件
git commit
git push
```

不得仅 commit 不 push。未推送到远程仓库的修改，不视为 ChatGPT 与 Codex 之间可交接的完成状态。

提交消息使用中文，建议格式：

```text
模块：简要说明本次修改

- 修改点一
- 修改点二
- 验证方式
```

不得把多个互不相关的任务合并成一个提交。

如果 `git push` 失败，必须在汇报中说明：

1. Push 失败原因。
2. 本地 commit hash。
3. 当前本地分支与远程分支状态。
4. 建议处理方式。

## 15. 汇报规范

Codex 完成任务后，必须按以下结构汇报：

```text
一、任务结论
说明任务是否完成。

二、主要修改
列出修改文件和修改内容。

三、实现说明
说明关键设计和实现方式。

四、验证结果
列出执行过的命令、测试、构建和结果。

五、影响范围
说明可能影响的模块、接口、数据或部署。

六、风险与遗留问题
说明未验证项、潜在风险和建议后续动作。

七、PROJECT_BASELINE.md 处理
说明是否需要更新当前阶段动态基线；如果已更新，列出更新内容；如果未更新，说明原因。

八、提交与推送信息
给出 commit hash、提交消息和 push 结果。
```

## 16. 禁止行为

以下行为默认禁止：

- 未同步远程代码直接修改。
- 未检查工作区状态直接覆盖。
- 未说明原因扩大任务范围。
- 未确认用途删除代码。
- 未验证就声称完成。
- 提交后不推送且不说明原因。
- 将密钥、Token、证书写入仓库。
- 把临时排查命令写成正式部署方案。
- 用大量代码绕开一个简单问题。
- Codex 声称已经通过 Prompt 自动切换了本次请求的模型、推理强度或响应速度。
- 在未读取 `AGENTS.md` 和 `PROJECT_BASELINE.md` 的情况下执行依赖项目边界或当前阶段状态的任务。
- 验收后当前阶段基线已经变化，却不更新、不建议更新、也不说明不更新原因。
