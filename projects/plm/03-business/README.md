# plm 业务规约

## 业务定位

本项目为产品研发及销售管理系统，基于 RuoYi / RuoYi-Vue MIT 开源框架进行二次开发。

软件著作权和业务资产边界应明确区分：

- RuoYi 原始框架代码和第三方开源组件不属于本项目新增业务成果。
- 本项目新增的 PLM 业务功能、业务模型、流程逻辑、接口服务、页面交互和相关文档属于本项目核心业务资产。

## 主要业务能力

当前已确认业务能力包括：

- 产品研发管理。
- 合同需求管理。
- 项目立项管理。
- 物料管理。
- 生产信息管理。
- 业务附件管理。
- 流程提交与审核。
- 基础权限管理。
- 系统日志与监控。

前端对应能力包括：

- 产品研发管理页面。
- 合同需求管理页面。
- 项目立项管理页面。
- 物料管理页面。
- 生产信息管理页面。
- 业务附件上传与预览。
- 流程提交、审核与状态展示。
- 权限菜单与用户操作界面。

## 业务建模原则

- 业务对象必须有清晰生命周期。
- 流程状态与业务状态必须区分。
- 流程提交、审核、状态展示必须与后端业务状态一致。
- 附件必须通过统一文件能力管理，不得在各业务模块分散实现文件访问逻辑。
- 权限菜单、用户操作界面和后端权限校验必须一致。
- 标准产品能力与客户定制能力应保持边界，避免定制逻辑污染核心模型。

## 文件与附件业务

PLM 业务文件包括但不限于：

- 合同附件。
- 项目附件。
- 图纸。
- 图片。
- 工艺文件。
- 生产相关附件。
- 流程附件。

所有业务文件统一存储于 MinIO。

文件预览必须通过统一 `FilePreview` 能力进入，业务页面不得直接依赖 PDF.js、ONLYOFFICE、MinIO URL 或文件后缀判断。

当前文件预览业务策略：

- 图片支持 `jpg/jpeg/png/gif/bmp/webp`。
- Office 支持 `doc/docx/xls/xlsx/ppt/pptx`。
- 扩展 Office 支持 `wps/wpt/et/ett/dps/dpt/txt/md/csv`。
- PDF 当前统一纳入 ONLYOFFICE 只读预览。
- `html/htm/xml/mht/mhtml`、Apple iWork、Visio、视频、音频、压缩包、CAD/数模图纸当前不开放。

## ONLYOFFICE 用户绑定规则

ONLYOFFICE 预览用户信息必须遵守：

- `editorConfig.user.id` 使用 `plm-user-{userId}`。
- `editorConfig.user.name` 使用 `nickName(userName)`。
- 昵称为空时使用 `userName`。
- 账号为空时使用“PLM用户”。
- 不使用手机号、邮箱、身份证等敏感信息作为 ONLYOFFICE 用户 ID。

## ONLYOFFICE 业务边界

当前 ONLYOFFICE 只用于只读预览。

禁止默认启用：

- 在线编辑。
- 保存回写。
- 协同编辑。
- 批注。
- 历史版本。

TXT/CSV 预览可通过 ONLYOFFICE Conversion API 即时转换为 DOCX/XLSX；转换产物不入库、不替换原文件。

## 权限业务

后续重点关注：

- 菜单权限。
- 角色权限。
- 数据权限。
- 项目权限。
- 流程权限。
- 附件权限。

权限不能只依赖前端隐藏按钮，必须以后端校验为准。
