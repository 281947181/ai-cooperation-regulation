# IoT 平台项目规约

## 1. 项目定位

本项目面向工业设备管理、设备数据采集、配置下发、流程管理和中心侧运维管理。

核心目标是形成可部署、可维护、可扩展的工业 IoT 管理平台，而不是一次性演示系统。

## 2. 默认技术栈

后端：

- Java 17。
- Spring Boot。
- RuoYi Fast 单体版本。
- MyBatis / MyBatis Plus 按项目实际情况确定。
- Flowable 用于业务流程。

前端：

- Vue 3。
- Element Plus。
- RuoYi Vue3 前端体系。

数据与中间件：

- MySQL 或 PostgreSQL 存储业务数据。
- TDengine 或其他时序库存储高频采集数据。
- Redis 用于缓存。
- MQTT 用于设备配置下发和消息交互。

部署：

- 统一 Docker Compose。
- 配置、数据、日志外置挂载。

## 3. 核心业务层级

设备管理默认层级：

```text
地区 / 厂区 / 厂房 / 产线（可选） / 设备
```

该层级是设备组织、权限过滤、配置下发、数据查询和统计分析的重要基础，不得随意简化为单层设备表。

## 4. 架构原则

- 优先保持 RuoYi 原有结构，不做无必要的框架级改造。
- 业务模块应清晰分层：Controller、Service、Mapper、Domain、DTO/VO。
- 配置下发应直接完善到 MQTT payload 下发，不设置没有实际价值的中间步骤。
- 流程能力通过 Flowable 集成，但业务状态不能完全依赖流程引擎黑盒。
- 高频采集数据与业务管理数据分库存储或至少分模型管理。

## 5. 数据原则

业务数据关注一致性、权限和可追溯。

时序数据关注写入吞吐、查询窗口、保留策略和压缩策略。

不得把高频时序数据直接塞入普通业务表。

## 6. 部署原则

所有服务通过统一 Docker Compose 部署。

Compose 应至少考虑：

- 后端服务。
- 前端服务或 Nginx。
- 数据库。
- Redis。
- MQTT Broker。
- TDengine 或时序库。
- 文件存储服务（如 MinIO，按项目需要）。

外部挂载建议分为：

```text
config/
data/
logs/
```

## 7. AI 开发约束

Codex 修改本项目时：

- 后端任务默认不读取前端目录，除非任务涉及页面。
- 前端任务默认不读取后端全量代码，除非需要接口联调。
- Flowable 相关任务必须同时考虑运行表、历史表、业务表之间的边界。
- Docker 相关任务必须避免破坏现有数据卷。
- TDengine 相关任务必须谨慎处理容器重启、数据目录、leader 恢复等问题。

## 8. 验收要求

后端修改至少要求：

- 编译通过或说明无法编译原因。
- 接口路径、权限标识、返回结构清楚。
- 数据库变更有 SQL 或迁移说明。

前端修改至少要求：

- 页面能构建或说明无法构建原因。
- 菜单、路由、权限标识匹配。
- 与后端接口字段一致。

部署修改至少要求：

- `docker compose config` 通过。
- 卷挂载、端口、环境变量说明清楚。

## 9. 已知重点风险

- Flowable 动态跳转、驳回、加签容易造成历史记录和业务状态不一致。
- TDengine 容器重启后可能出现恢复、leader、数据目录权限等问题。
- RuoYi 权限标识、菜单、接口映射需要保持一致。
- 设备层级和数据权限设计不能后期临时拼接。
