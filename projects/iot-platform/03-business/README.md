# iot-platform 业务规约

## 核心业务定位

本项目围绕工业设备管理、数据采集、配置下发、流程审批和中心侧运维展开。

业务模型必须服务于长期运维，而不是只满足单次页面展示。

## 设备组织层级

设备管理默认层级：

```text
地区 / 厂区 / 厂房 / 产线（可选） / 设备
```

该层级是以下能力的基础：

- 设备组织。
- 数据权限过滤。
- 配置下发范围选择。
- 采集数据查询。
- 统计分析。
- 运维责任划分。

不得随意简化为单层设备表。

## 设备核心信息

设备模型至少应考虑：

- 设备唯一标识。
- 设备名称。
- 设备类型。
- 所属组织层级。
- 通讯参数。
- 采集点位。
- 当前状态。
- 生命周期状态。
- 维护责任人。

## 配置下发

配置下发必须形成明确的 MQTT payload。

配置应具备：

- 下发对象。
- 配置版本。
- payload 内容。
- 下发时间。
- 下发结果。
- 失败原因。
- 操作人。

不得只保存最新配置而丢失历史。

## 流程业务

Flowable 可用于维修、保养、审批等流程。

业务设计必须区分：

- 流程实例状态。
- 业务单据状态。
- 用户任务状态。
- 审批结论。

动态驳回、跳转、加签等能力必须谨慎处理历史记录与业务状态一致性。

## 权限与数据范围

设备、流程、配置、数据查询均应考虑数据范围。

常见维度：

- 角色。
- 部门。
- 工厂。
- 区域。
- 设备归属。

权限不能只依赖前端隐藏按钮。
