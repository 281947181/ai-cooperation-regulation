# iot-platform 架构规约

## 总体架构

本项目默认采用 RuoYi Fast 单体后端 + Vue3 前端 + Docker Compose 部署模式。

架构目标：

- 优先保持 RuoYi 原有结构，不做无必要的框架级改造。
- 业务模块清晰分层。
- 高频采集数据与业务管理数据分离建模。
- 流程引擎服务于业务流转，但不替代业务状态模型。
- 配置下发直接形成 MQTT payload，不设置没有价值的中间步骤。

## 后端分层

默认分层：

```text
Controller
Service
Mapper
Domain / Entity
DTO / VO
```

约束：

- Controller 只处理接口入参、权限、返回结构，不承载复杂业务逻辑。
- Service 承载业务编排和事务边界。
- Mapper 只负责数据访问。
- DTO/VO 必须与接口语义匹配，不直接暴露不适合前端使用的内部实体。

## 前端边界

前端基于 Vue3 + Element Plus + RuoYi 前端体系。

约束：

- 页面逻辑不得无限堆叠在单个组件内。
- 菜单、路由、权限标识必须与后端保持一致。
- 前端字段必须与后端 DTO/VO 对齐。

## 数据边界

业务数据关注：

- 一致性。
- 权限控制。
- 可追溯。
- 可审计。

时序数据关注：

- 写入吞吐。
- 查询窗口。
- 保留策略。
- 压缩策略。

禁止把高频时序数据直接塞入普通业务表。

## 流程边界

Flowable 用于业务流程流转。

约束：

- 业务状态不能完全依赖 Flowable 黑盒运行状态。
- 动态跳转、驳回、加签必须同时考虑运行表、历史表和业务表一致性。
- 任务候选人、办理人、业务状态应有明确归属。

## 消息与配置下发

MQTT 用于设备配置下发和消息交互。

配置下发应直接形成可执行 payload，并记录版本、下发对象、下发结果和审计信息。

## 部署边界

统一使用 Docker Compose。

Compose 至少考虑：

- 后端服务。
- 前端或 Nginx。
- 数据库。
- Redis。
- MQTT Broker。
- TDengine 或时序库。
- MinIO 或其他文件存储（按需求）。

配置、数据、日志必须外置。应用源码、静态资源、脚本等能外挂的内容优先外挂，减少无必要镜像重建。
