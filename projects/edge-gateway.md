# 边缘网关与中心侧配置平台项目规约

## 1. 项目定位

本项目面向工业现场边缘网关、协议适配、中心侧配置管理、配置下发、数据采集、清洗、存储和上报。

核心目标是建立稳定、可部署、可维护的边缘到中心的数据链路。

## 2. 默认技术栈

后端：

- Java 17。
- Spring Boot 稳定版本。
- PostgreSQL 存储中心侧业务配置。
- Kafka 或 MQTT 承载消息链路，按场景区分。

边缘侧：

- Java 或轻量服务。
- OPC UA、Modbus TCP、S7 等协议适配。
- 配置驱动采集。

部署：

- Docker Compose。
- 配置、数据、日志外置。

## 3. 架构原则

- 中心侧负责配置管理、下发、监控、数据消费和治理。
- 边缘侧负责协议连接、采集执行、本地缓存、断点续传和状态上报。
- 控制类命令优先同步直连，消息队列主要用于数据上报和异步事件。
- 协议适配必须与业务模型解耦。
- 点位模型必须可版本化、可下发、可追溯。

## 4. 协议适配原则

OPC UA：

- 优先使用成熟 SDK。
- 关注 endpoint、证书、session、subscription 生命周期。
- 避免一点一连接或一设备一线程的低效设计。

Modbus TCP / S7：

- 默认轮询模型。
- 点位合并读取。
- 控制并发和超时。
- 统一错误码和状态上报。

## 5. 配置下发原则

配置必须包含：

- 网关标识。
- 设备标识。
- 协议类型。
- 连接参数。
- 点位定义。
- 采集周期。
- 数据类型。
- 上报策略。
- 版本号。

配置下发必须可审计，不得只保留最新状态而丢失历史。

## 6. 数据原则

采集数据应区分：

- 原始值。
- 清洗值。
- 质量码。
- 时间戳。
- 设备标识。
- 点位标识。
- 网关标识。

中心侧不能假设所有边缘数据都实时、完整、顺序到达。

## 7. AI 开发约束

Codex 修改本项目时：

- 不得把协议连接逻辑写死在业务 Controller 中。
- 不得为每个点位创建独立线程。
- 不得绕过配置版本管理直接下发临时配置。
- 日志必须包含网关、设备、点位等关键定位信息。
- 网络异常、认证异常、协议异常必须分类处理。

## 8. 验收要求

至少验证：

- 配置能保存。
- 配置能下发。
- 边缘侧能解析配置。
- 协议连接异常能上报。
- 采集数据能进入预期链路。
- Docker Compose 配置可校验。

## 9. 已知重点风险

- 协议 SDK 版本差异导致 API 不兼容。
- OPC UA endpoint 主机名、证书、session 问题容易造成连接失败。
- 工业网络不稳定，必须设计重连和降级策略。
- 点位规模扩大后，线程模型和连接模型可能成为瓶颈。
